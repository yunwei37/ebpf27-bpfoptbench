\title{\sys: A Benchmark for Agentic Optimization of Existing eBPF Programs}

\author{Anonymous Authors}
\affiliation{\institution{Anonymous Institution}}
\renewcommand{\shortauthors}{Anonymous}

\begin{abstract}
Kernel extensions such as eBPF are widely deployed on performance-critical
paths for networking, observability, and security. Optimizing these programs
requires cross-layer expertise—verifier semantics, JIT behavior, and workload
characteristics—and offers no reliable local proxy: the only trustworthy signal
comes from running the full pipeline. This makes eBPF optimization an iterative
search problem where each candidate must be executed to evaluate, a natural fit
for LLM agents. We present \sys, an auto-research framework and benchmark for
eBPF optimization where agents autonomously explore optimization strategies
under real execution feedback. Agents propose actions, the framework executes
real applications under real workloads, and structured observations (verifier
status, JIT output, runtime counters) guide the next iteration. \sys covers 6
production applications (146 programs), exposes action spaces from suite-wide
pass selection to per-program policy, and enforces anti-gaming rules with
hidden correctness checks. Preliminary data motivates autonomous exploration:
static policies yield a near-flat 1.004$\times$ geomean, and local rewrite
counts do not predict speedups—programs with bytecode changes regress
(1.032$\times$) while unchanged programs improve (0.983$\times$).
\end{abstract}

\keywords{eBPF, agents, optimization, benchmark, verifier, JIT}

\section{Background}

Prior eBPF optimization work shows real headroom: K2 optimizes bytecode before
load~\cite{xu2021synthesizing}; Merlin spans LLVM IR and
bytecode~\cite{mao2024merlin}; Kgent explores LLM-based kernel extension
generation~\cite{zheng2024kgent}. These answer how to optimize or generate
eBPF, but not whether an \emph{agent} can tune existing programs when the only
trustworthy feedback is the real verifier, JIT, and workload.

Agent benchmarks such as SWE-bench and
\(\tau\)-bench~\cite{swebench2023,taubench2024} evaluate agents with hidden
tests, but eBPF needs an oracle composing verifier acceptance, application
behavior, workload semantics, and performance measurement. A verifier-accepted
candidate can still break the app; a workload-correct candidate may only chase
noise. \sys fills this gap with a benchmark for closed-loop eBPF optimization.

\section{Motivation}

\sys models eBPF optimization as layered artifact transforms (Table~\ref{tab:layers}).
The first track focuses on pre-load bytecode, live bytecode/ReJIT, and optional
kernel/JIT transforms---layers exercisable with existing application loaders.
Action levels range from A0 (no-op) through A1 (suite-wide pass list), A2
(per-app), A3 (per-program), to A5 (profile-guided). The first paper targets
A1--A3.

\section{Benchmark Design}

\sys defines success as choosing an optimization action that survives the real
kernel and improves a real workload. Figure~\ref{fig:loop} shows the closed
loop: observe, propose action, execute real app, score.

\noindent\textbf{Task model.} A task specifies: (1) \emph{subject}---eBPF app,
workload, architecture; (2) \emph{action space}---allowed layer/granularity;
(3) \emph{budget}---attempts, time; (4) \emph{feedback}---raw artifacts and
summaries; (5) \emph{oracle}---verifier acceptance, app lifecycle, workload
correctness, performance outcome.

\noindent\textbf{Oracle.} A candidate succeeds only if: the kernel accepts it
through the real verifier; the app completes the benchmark lifecycle; workload
checks pass; performance beats the noise gate; and no forbidden shortcut was
used.

\noindent\textbf{Anti-gaming rules.} Forbidden: reducing workload duration,
filtering hard programs, hiding ReJIT failures, replacing app loaders. The rate
of invalidation attempts is itself a diagnostic metric.

\section{Preliminary Evaluation}

We structure evaluation as a benchmark-artifact study first, agent-performance
study second. \textbf{RQ1}: Why is eBPF optimization a hard closed-loop
problem, not static pass selection? \textbf{RQ2}: Does \sys expose verifier,
app, workload, and performance failures separately? \textbf{RQ3}: Do
closed-loop agents outperform no-op, static, random, and human-tuned baselines?

\noindent\textbf{Metrics.} Validity rate, workload success rate, performance
success rate, per-program geomean, regret to best-known action, cost (attempts,
time, tokens), integrity failure rate.

\noindent\textbf{Baselines.} No-op, noop-ReJIT (measurement perturbation
control), static policy, random/grid search, human-tuned policy, oracle
best-known (regret only).

\noindent\textbf{Current status.} The v3 substrate completes 6 production apps
(\texttt{bcc}, \texttt{tracee}, \texttt{tetragon}, \texttt{katran},
\texttt{cilium}, \texttt{otelcol-ebpf-profiler}) through native-loader paths
and records per-app raw payloads. Preliminary data (Table~\ref{tab:motivation})
already shows that static policies are noisy and rewrite counts do not predict
speedup. A workshop version need not show agents solve eBPF optimization---it
must make the task measurable and falsifiable. Next step: freeze 20--50 tasks
and run a scoreboard comparing no-op, static, random, human-tuned, and
LLM-agent policies.
