\title{\sys: An Auto-Research Framework and Benchmark for Agentic Kernel Extension Optimization}

\author{Anonymous Authors}
\affiliation{\institution{Anonymous Institution}}
\renewcommand{\shortauthors}{Anonymous}

\begin{abstract}
Kernel extensions such as eBPF are widely deployed on performance-critical paths.
% eBPF 等内核扩展被广泛部署在性能关键路径上。
Yet, optimizing these programs is hard because the space spans multiple layers including source code, LLVM backend, and kernel JIT. Static analysis cannot predict performance, and the same LLVM pass can speed up one program by 1.9$\times$ while slowing another by 14\%.
% 优化这些程序很难，因为空间跨越多层（源码、LLVM 后端、内核 JIT），静态分析无法预测性能，同一个 LLVM pass 可以让一个程序加速 1.9 倍却让另一个减速 14%。
We formulate eBPF optimization as an iterative search problem where LLM agents can leverage semantic information, cross-domain knowledge, and adaptive strategies to outperform traditional methods.
% 我们将 eBPF 优化框定为一个迭代搜索问题，LLM agents 可以利用语义反馈、跨领域知识和自适应策略来超越传统方法。
We present \sys, the first eBPF optimization auto-research framework, enabling safe isolation for kernel modifications, iterative exploration with bounded action spaces, and structured feedback across the compilation pipeline and exploration history.
% 我们提出 \sys，第一个 eBPF 优化自动研究框架，支持内核修改的安全隔离、有界动作空间的迭代探索、以及贯穿编译管线和探索历史的结构化反馈。
We instantiate \sys as a benchmark covering 6 production applications with 146 programs and 42 micro benchmarks, and enforce hack detection with hidden correctness checks.
% 我们将 \sys 实例化为一个 benchmark，涵盖 6 个生产应用的 146 个程序和 42 个微基准测试，并通过隐藏的正确性检查执行作弊检测。
Agent-guided exploration discovers optimizations at each compilation layer, with the best results being Cilium +12\% at source, OTEL +34\% at LLVM, and Tetragon +17\% at kernel JIT, while micro benchmarks achieve 21\% speedup on both x86 and ARM64.
% agent 引导的探索在每个编译层都发现了优化，最佳结果：源码层 Cilium +12%，LLVM 层 OTEL +34%，内核 JIT 层 Tetragon +17%，微基准在 x86 和 ARM64 上均达到 21% 加速。
\end{abstract}

\section{Background}

\paragraph{eBPF pipeline.}
eBPF is a kernel extension mechanism that runs programs on performance-critical paths for networking~\cite{cilium}, security~\cite{tetragon,falco}, and observability~\cite{bcc}.
% eBPF 是一种内核扩展机制，在网络、安全和可观测性等性能关键路径上运行程序。
Developers write eBPF programs in C or Rust, compile them with Clang/LLVM to BPF bytecode, and load them into the kernel where a verifier checks safety before JIT compilation to native code.
% 开发者用 C 或 Rust 编写 eBPF 程序，用 Clang/LLVM 编译成 BPF 字节码，加载到内核中，由 verifier 检查安全性后 JIT 编译成原生代码。
Each stage offers optimization opportunities, including source-level restructuring, LLVM backend tuning, and kernel JIT lowering with architecture-specific instructions.
% 每个阶段都提供优化机会，包括源码级重构、LLVM 后端调优、以及使用架构特定指令的内核 JIT 降级。

\paragraph{LLM agents and auto-research frameworks.}
LLM agents reason about tasks and execute actions through tool calls~\cite{swebench2024,llmagentsesurvey2025}, and auto-research frameworks extend this paradigm to autonomous experimentation~\cite{gso2025}.
% LLM agents 通过工具调用来推理任务并执行动作，自动研究框架将这一范式扩展到自主实验。
Recent work has applied these frameworks to C++ algorithm optimization~\cite{pie2024}, loop scheduling~\cite{compilot2025,awarecompiler2025}, and GPU kernel generation~\cite{kernelbench2025,gso2025}.
% 最近的工作已将这些框架应用于 C++ 算法优化、循环调度和 GPU kernel 生成。

\section{Motivation}

\paragraph{eBPF optimization is hard.}
The search space is large because eBPF programs span multiple optimization layers from source restructuring through LLVM backend tuning to kernel JIT lowering, each with architecture-specific choices.
% 搜索空间很大，因为 eBPF 程序跨越多个优化层，从源码重构到 LLVM 后端调优再到内核 JIT 降级，每层都有架构特定的选择。
Static analysis cannot predict performance because bytecode size does not predict JIT output, JIT size does not predict runtime, and microbenchmark gains may not transfer to real workloads.
% 静态分析无法预测性能，因为字节码大小无法预测 JIT 输出，JIT 大小无法预测运行时，微基准的收益可能无法迁移到真实工作负载。
Context determines effects, so a transformation profitable on x86 may regress on arm64, and the same pass that speeds up one program slows down another.
% 上下文决定效果，因此在 x86 上有收益的变换可能在 arm64 上回退，同一个 pass 加速一个程序却减速另一个。
Table~\ref{tab:motivation} shows this concretely, as a single pass can speed up one program by 1.9$\times$ while slowing another by 14\%, so the only trustworthy signal comes from execution.
% 表 1 具体展示了这一点，同一个 pass 可以让一个程序加速 1.9 倍，却让另一个程序减速 14%，因此唯一可信的信号来自执行。

\paragraph{Existing eBPF optimization.}
Existing tools take fixed approaches. K2~\cite{xu2021synthesizing} applies stochastic superoptimization to BPF bytecode as a one-shot optimizer, Merlin~\cite{mao2024merlin} combines LLVM IR and bytecode transformations but requires source access at compile time, and the LLVM BPF backend provides static peephole optimizations with uniform heuristics.
% 现有工具采用固定方法。K2 作为一次性优化器对 BPF 字节码应用随机超优化，Merlin 结合 LLVM IR 和字节码变换但需要编译时源码访问，LLVM BPF 后端提供统一启发式的静态窥孔优化。
These approaches all search within predefined transformation spaces, but none can understand why a transformation succeeds, generalize patterns across programs, or devise new optimization strategies.
% 这些方法都在预定义的变换空间内搜索，但没有一个能理解变换为何成功、跨程序归纳规律、或设计新的优化策略。

\section{Framework Design}

\paragraph{Goals.}
\sys enables agents to iteratively explore eBPF optimization strategies under real execution feedback.
% \sys 使 agents 能够在真实执行反馈下迭代探索 eBPF 优化策略。
The framework addresses four goals: (1) crash isolation so failed optimizations do not corrupt the host, (2) bounded action spaces that make search tractable and prevent reward hacking, (3) multi-layer feedback from verifier, JIT, and runtime that is persisted for querying and iterative refinement, and (4) a closed loop that drives agent exploration under budget constraints.
% 框架解决四个目标：(1) 崩溃隔离使失败的优化不会破坏主机，(2) 有界动作空间使搜索可行并防止作弊，(3) 来自 verifier、JIT 和运行时的多层反馈，持久化以供查询和迭代改进，(4) 驱动 agent 在预算约束下进行循环探索。

\paragraph{Architecture.}
Figure~\ref{fig:arch} shows the three-layer system architecture.
% 图 1 展示了三层系统架构。
Agents select optimization strategies, the framework executes them in Docker containers deployed to KVM or AWS VMs for kernel-level crash recovery, and structured feedback guides the next iteration.
% agents 选择优化策略，框架在部署到 KVM 或 AWS VM 的 Docker 容器中执行以提供内核级崩溃恢复，结构化反馈指导下一次迭代。
Optimization spans three compilation layers, from source restructuring through LLVM backend tuning to kernel JIT lowering, with the kernel layer using Kops~\cite{kops2026} to safely extend the eBPF JIT with native operations.
% 优化跨越三个编译层，从源码重构到 LLVM 后端调优再到内核 JIT 降级，内核层使用 Kops 安全扩展 eBPF JIT 以支持原生操作。
The framework supports live bytecode optimization leveraging runtime information such as loaded maps and counters.
% 框架支持利用已加载 map 和计数器等运行时信息的运行时字节码优化。

\paragraph{Iteration.}
Each iteration follows a closed loop in which agents submit an optimization configuration, the framework executes it under budget constraints, and returns structured feedback including verifier logs, JIT output size, and per-program runtime counters.
% 每次迭代遵循闭环，其中 agents 提交优化配置，框架在预算约束下执行，并返回包括 verifier 日志、JIT 输出大小和每程序运行时计数器的结构化反馈。
The framework persists all session history for cross-iteration learning, and enforces integrity constraints via OS-level policy~\cite{actplane2026}: agents interact only through \texttt{make} targets and cannot modify benchmark code, reduce workload duration, or fabricate results. 
% 框架持久化所有 session 历史以支持跨迭代学习，并强制执行完整性约束使得 agents 不能减少工作负载时长、从测量中过滤程序或伪造结果。

\section{Benchmark Design}

\paragraph{Workloads.}
The benchmark covers 6 production applications with 146 BPF programs.
% benchmark 涵盖 6 个生产应用，共 146 个 BPF 程序。
Networking apps (Cilium, Katran) use kernel pktgen~\cite{pktgen} to stress the datapath, while security and tracing apps (Tetragon, Tracee, BCC) use stress-ng~\cite{stressng} to generate high-rate syscall workloads, and OTEL uses interpreter hash loops for continuous profiling.
% 网络应用（Cilium, Katran）使用内核 pktgen 压测数据路径，安全和追踪应用（Tetragon, Tracee, BCC）使用 stress-ng 生成高频系统调用负载，OTEL 使用解释器哈希循环进行持续 profiling。
Additionally, 42 microbenchmarks isolate pure-bytecode execution and helper/map access under \texttt{BPF\_PROG\_TEST\_RUN}.
% 另有 42 个微基准在 BPF\_PROG\_TEST\_RUN 下隔离纯字节码执行和 helper/map 访问。
All comes with hidden correctness unittest to prevent reward hacking.
% 全部通过隐藏的正确性检查执行作弊检测.

\paragraph{Metrics.}
The benchmark measures both BPF-level and application-level impact.
% benchmark 同时测量 BPF 级和应用级影响。
BPF counters report per-program \texttt{run\_cnt\_delta} and \texttt{run\_time\_ns\_delta} between baseline and post-optimization phases, enabling per-program speedup analysis.
% BPF 计数器报告基线和优化后阶段之间的每程序 run\_cnt\_delta 和 run\_time\_ns\_delta，支持每程序加速分析。
Application metrics capture workload throughput with BPF stats disabled to avoid measurement overhead.
% 应用指标在禁用 BPF 统计的情况下捕获工作负载吞吐量以避免测量开销。
Runs use x86 KVM and arm64 AWS to expose architecture-dependent effects.
% 运行使用 x86 KVM 和 arm64 AWS 以暴露架构相关效应。

\section{Preliminary Evaluation}

\paragraph{Multi-layer optimization results.}
Figure~\ref{fig:layers} shows agent-discovered optimizations across source, LLVM/bytecode, and kernel JIT layers, with the best results being Cilium +12\% at source, OTEL +34\% at LLVM, and Tetragon +17\% at kernel JIT.
% 图展示了每层的最佳逐应用结果。

\paragraph{Agent exploration process.}
Using Claude Code~\cite{claudecode2025} (Opus 4.5) and Codex~\cite{openaicodex2025} (GPT-5.5) on x86 KVM (8 vCPUs, 64GB) and ARM64 AWS t4g.small, LLM agents autonomously drove 2,640 benchmark sessions over 7 weeks (May 6 -- Jun 26, 2026), consuming 445M + 531M tokens with avg 7h per session (longest 19h), with human involvement limited to framework design (10.7K lines of Python) and triggering exploration, while optimization algorithm design was fully autonomous.
% 使用 Claude Code 和 Codex 在 x86 KVM (8 vCPUs, 64GB) 和 ARM64 AWS t4g.small 上，LLM agents 在 7 周内（2026年5月6日-6月26日）自主驱动了 2,640 个 benchmark session，消耗 445M + 531M tokens，平均每 session 7h（最长 19h），人工参与仅限于框架设计和触发探索，算法设计全程无人工指导。
