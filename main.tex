\title{\sys: A Benchmark for Agentic Optimization of Existing eBPF Programs}

\author{Anonymous Authors}
\affiliation{\institution{Anonymous Institution}}
\renewcommand{\shortauthors}{Anonymous}

\begin{abstract}
Kernel extensions such as eBPF are widely deployed on performance-critical
paths for networking, observability, and security. Optimizing these programs
requires cross-layer expertise—verifier semantics, JIT behavior, and workload
characteristics—and offers no reliable local proxy: the only trustworthy signal
comes from running the full pipeline. This makes eBPF optimization an iterative
search problem where each candidate must be executed to evaluate, a natural fit
for LLM agents. We present \sys, an auto-research framework and benchmark for
eBPF optimization where agents autonomously explore optimization strategies
under real execution feedback. Agents propose actions, the framework executes
real applications under real workloads, and structured observations (verifier
status, JIT output, runtime counters) guide the next iteration. \sys covers 6
production applications (146 programs), exposes action spaces from suite-wide
pass selection to per-program policy, and enforces anti-gaming rules with
hidden correctness checks. Preliminary data motivates autonomous exploration:
static policies yield a near-flat 1.004$\times$ geomean, and local rewrite
counts do not predict speedups—programs with bytecode changes regress
(1.032$\times$) while unchanged programs improve (0.983$\times$).
\end{abstract}

\section{Background}

\paragraph{eBPF pipeline.} eBPF programs run on performance-critical paths for networking, observability,
and security. Each program passes through a multi-stage pipeline: compilation
from source to bytecode, verification by the in-kernel verifier, JIT
compilation to native code, and execution under real workloads. The verifier
enforces safety properties; the JIT lowers bytecode to architecture-specific
machine code; the workload determines which code paths are hot.

\paragraph{LLM agents.} LLM agents are increasingly applied to code optimization tasks.
KernelBench~\cite{kernelbench2025} evaluates LLMs on GPU kernel generation with
execution feedback. ComPilot~\cite{compilot2025} establishes a closed-loop
interaction between LLMs and compilers for loop optimization.
GSO~\cite{gso2025} benchmarks SWE-agents on challenging software optimization
tasks across real codebases. These works demonstrate that agents can leverage
execution feedback for iterative code improvement.

\section{Motivation}

\noindent\textbf{Why is eBPF optimization hard?} Optimizing eBPF programs
requires cross-layer expertise: verifier semantics, JIT behavior across
architectures, and workload characteristics. Unlike traditional compiler
optimization, there is no reliable local proxy—bytecode size does not predict
JIT output, JIT size does not predict runtime, and microbenchmark gains may not
transfer to real workloads. The only trustworthy signal comes from running the
full pipeline.

\noindent\textbf{Existing eBPF optimization.} K2~\cite{xu2021synthesizing}
applies stochastic search to optimize BPF bytecode before load.
Merlin~\cite{mao2024merlin} combines LLVM IR and bytecode transformations. The
LLVM BPF backend includes static peephole optimizations. These approaches use
fixed heuristics or search strategies without adapting to execution feedback.

\noindent\textbf{Gap.} Existing agent benchmarks (KernelBench, ComPilot, GSO)
do not cover eBPF's unique verifier/JIT/workload loop. Existing eBPF optimizers
(K2, Merlin, LLVM) do not evaluate whether agents can learn effective
strategies from execution feedback.

\noindent\textbf{Evidence.} Table~\ref{tab:motivation} summarizes data from our
corpus. Static policies yield near-flat results; local rewrite counts do not
predict speedups. This motivates an auto-research framework where agents
autonomously explore strategies under real feedback.

\section{Framework Design}

optimization strategies. Figure~\ref{fig:loop} shows the closed loop: agents

\noindent\textbf{Task model.} A task specifies: (1) \emph{subject}—eBPF app,
workload, architecture; (2) \emph{action space}—allowed layer/granularity;
(3) \emph{budget}—attempts, time; (4) \emph{feedback}—verifier status, JIT
output, runtime counters; (5) \emph{oracle}—composed correctness and
performance checks.

\noindent\textbf{Oracle.} A candidate succeeds only if: the verifier accepts
it; the app completes the benchmark lifecycle; workload checks pass;
performance beats the noise gate; and no forbidden shortcut was used.

\noindent\textbf{Anti-gaming rules.} Forbidden: reducing workload duration,
filtering programs, hiding failures, replacing loaders.

\section{Preliminary Evaluation}

\noindent\textbf{Research questions.} RQ1: Is eBPF optimization a hard
closed-loop problem? RQ2: Does \sys expose failures separately? RQ3: Do agents
outperform static baselines?

\noindent\textbf{Metrics.} Validity rate, workload success rate, performance
geomean, regret, cost (attempts, tokens), integrity failure rate.

\noindent\textbf{Baselines.} No-op, noop-ReJIT, static policy, random search,
human-tuned policy.

\noindent\textbf{Status.} The framework completes 6 production apps (bcc,
tracee, tetragon, katran, cilium, otelcol-ebpf-profiler) through native-loader
paths. Table~\ref{tab:motivation} shows static policies are insufficient. Next:
freeze 20--50 tasks and compare agent policies.
