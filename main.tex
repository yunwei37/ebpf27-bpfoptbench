\title{\sys: An Auto-Research Framework and Benchmark for Agentic eBPF Optimization}

\author{Anonymous Authors}
\affiliation{\institution{Anonymous Institution}}
\renewcommand{\shortauthors}{Anonymous}

\begin{abstract}
Kernel extensions such as eBPF are widely deployed on performance-critical
paths. Optimizing these programs is hard: the search space spans multiple
layers, static analysis cannot predict performance, and context determines
whether a transformation helps or hurts. This makes eBPF optimization an iterative search problem where
LLM agents can leverage semantic feedback, cross-domain knowledge, and
adaptive strategies to outperform traditional methods. We present \sys, an
auto-research framework providing safe iteration with bounded action spaces,
crash isolation, and structured feedback from verifier logs, JIT output, and
runtime counters. We instantiate \sys as a benchmark covering 6 production
applications with 146 programs and 42 micro benchmarks, and enforce
anti-gaming rules with hidden correctness checks. Agent-guided exploration
across 2,874 sessions discovered that per-program tuning is essential, as
the same pass speeds up some programs by 1.9$\times$ while slowing others
by 14\%. The tuned kinsn policy achieves 0.94$\times$ BPF runtime (6\%
speedup) on x86 and 1.21$\times$ (21\% speedup) on ARM64 micro benchmarks.
\end{abstract}

\section{Background}

\paragraph{eBPF pipeline.}
eBPF is a kernel extension mechanism that runs programs on
performance-critical paths for networking~\cite{cilium},
security~\cite{tetragon,falco}, and observability~\cite{bcc}. Developers
write eBPF programs in C or Rust, compile them with Clang/LLVM to BPF
bytecode, and load them into the kernel where a verifier checks safety
before JIT compilation to native code. Each stage offers optimization
opportunities: source-level restructuring, LLVM backend tuning, and
kernel JIT lowering with architecture-specific instructions.

\paragraph{LLM agents and auto-research frameworks.}
LLM agents use LLMs to reason about what to do and take actions via tool
calls~\cite{swebench2023,llmagentsesurvey2025}. Auto-research frameworks enable
agents to conduct experiments autonomously~\cite{gso2025}. Recent work applies this to C++ algorithm
optimization~\cite{pie2024}, loop scheduling~\cite{compilot2025,awarecompiler2025},
and GPU kernel generation~\cite{kernelbench2025,gso2025}.

\section{Motivation}

\paragraph{eBPF optimization is hard.}
The search space is large: eBPF programs span multiple optimization layers
from source restructuring through LLVM backend tuning to kernel JIT lowering,
each with architecture-specific choices. Static analysis cannot predict
performance because bytecode size does not predict JIT output, JIT size does
not predict runtime, and microbenchmark gains may not transfer to real
workloads. Context determines effects: a transformation profitable on x86
may regress on arm64, and the same pass that speeds up one program slows down
another. Table~\ref{tab:motivation} shows concrete evidence: the bulk\_memory
pass achieves 1.9$\times$ speedup on \texttt{siphash\_rotate64\_mixer} but
causes 14\% regression on \texttt{bpftrace\_comm\_key\_fnv}. The only
trustworthy signal comes from execution.

\paragraph{Existing eBPF optimization.}
K2~\cite{xu2021synthesizing} applies stochastic superoptimization to BPF
bytecode but operates as a one-shot optimizer without iterative feedback.
Merlin~\cite{mao2024merlin} combines LLVM IR and bytecode transformations
but requires source access and operates at compile time. The LLVM BPF backend
includes static peephole optimizations but these are fixed heuristics applied
uniformly. These approaches use fixed search strategies, and none adapts
policy based on execution feedback.

\section{Framework Design}

\paragraph{Goals.}
\sys enables agents to iteratively explore eBPF optimization strategies under
real execution feedback. The framework addresses four goals: (1) crash
isolation so failed optimizations do not corrupt the host, (2) bounded action
spaces to make search tractable, (3) structured semantic feedback beyond
pass/fail signals, and (4) reproducible execution on real workloads.

\paragraph{Architecture.}
Figure~\ref{fig:arch} shows the closed loop. LLM agents select optimization
strategies across three layers: source restructuring, LLVM backend tuning, and
kernel JIT lowering. The framework packages applications into Docker images,
deploys to KVM or AWS for isolated execution, and returns structured feedback
including verifier logs, JIT output size, and per-program runtime counters. The
first benchmark track focuses on live bytecode optimization: pass list
selection, per-program policies, and ReJIT after programs are loaded,
leveraging runtime information such as loaded maps and verifier states.

\paragraph{Action space.}
\sys defines action granularity from coarse to fine. A0 (no-op) leaves all
programs unchanged as baseline. A1 (suite-wide) applies a fixed pass list to
all programs. A2 (per-app) applies different policies to different
applications. A3 (per-program) targets individual programs, such as optimizing
only high-run-count programs. Higher levels A4 (per-site) and A5
(profile-guided) control individual transformation sites and use PMU data. The
first benchmark track targets A1--A3; higher levels require stable per-site
reporting infrastructure.

\section{Benchmark Design}

\paragraph{Goals.}
The benchmark evaluates agent actions with three goals: (1) production coverage
through 6 real applications with 146 programs and 42 micro benchmarks, (2)
anti-gaming enforcement through hidden correctness checks, and (3) fair
comparison through noise-floor baselines.

\paragraph{Workloads.}
The benchmark covers Cilium (CNI networking), Tetragon (security observability),
Tracee (runtime security), Katran (load balancing), BCC (tracing tools), and
OTEL (continuous profiling). Each task specifies a subject (application,
workload, architecture), action space (optimization layer and granularity),
budget (attempts, wall-clock time, tokens), and feedback (verifier logs, JIT
size, per-program \texttt{run\_cnt\_delta} and \texttt{run\_time\_ns\_delta}).
Production applications with representative workloads ensure results transfer
to real deployments.

\paragraph{Evaluation and metrics.}
A candidate succeeds only if it passes all evaluation gates: verifier
acceptance, application lifecycle completion (startup, attach, workload,
shutdown), application-specific semantic checks (packet forwarding for Cilium,
event detection for Tetragon, load balancing for Katran), and performance
improvement beyond the noise gate (0.04$\times$ based on no-op floor).
Forbidden actions include reducing workload duration, modifying inputs/outputs,
filtering programs from measurement, hiding ReJIT failures, replacing loaders,
and fabricating results. \sys reports validity rate, workload success rate,
performance success rate, geomean ratio (per-program geometric mean of
post/baseline), regret (gap to best-known action), cost (attempts, time,
tokens), and integrity failure rate.

\section{Preliminary Evaluation}

\paragraph{Exploration scale.}
We conducted 2,874 benchmark sessions exploring 25 unique pass configurations
across 6 production applications. The exploration was driven by LLM agents
(Claude Code and Codex), with 1,651 commits from automated agents. Figure~\ref{fig:timeline}
shows the exploration timeline, with key discoveries annotated.

\paragraph{Kinsn optimization results.}
The agent-guided exploration converged on kinsn (kernel instruction lowering)
as the most effective optimization family, applying 305,449 transformation
sites. On x86, the tuned kinsn policy achieves 0.94$\times$ BPF runtime
(6\% speedup) across 6 apps. On ARM64 micro benchmarks, kinsn achieves
1.21$\times$ speedup (21\% faster) with 24 wins out of 29 benchmarks.
Table~\ref{tab:results} summarizes the per-app breakdown.

\paragraph{Key findings.}
The exploration revealed that per-program tuning is essential: the same kinsn
pass that speeds up tracee by 7\% causes OTEL to regress by 7\%. Static
policies cannot capture this variance. The framework enabled rapid iteration,
with 515 sessions in 4 days during the intensive exploration period.
